FleetRTS — combat layer design notes
====================================

Original shape
--------------
The playable prototype grew as a single-player campaign loop: JSON-driven ships,
one human “player” side, an “enemy” side, strike vs recovery rounds, then debrief.

Why treat combat as a sandbox
------------------------------
Multiplayer (co-op, custom lobbies, maybe PvP) and “custom games” are easier if the
combat layer stays **data-driven and side-agnostic** where possible:

  - **Sides** — Today most logic hard-codes `"player"` vs `"enemy"`. Longer term,
    “team id” or faction slots allow co-op (multiple player teams vs AI), PvP, or
    three-way fights without rewriting the whole sim.

  - **Pressure / difficulty** — `enemy_pressure` (and similar knobs) are examples
    of **scenario parameters** hosts can set instead of fixed campaign curves.

  - **Objectives** — Strike / recovery / pods are one ruleset. A sandbox-friendly
    `MissionState` (or scenario descriptor) can later swap objectives (hold zone,
    escort, king of the hill) while reusing movement, weapons, and fog.

  - **Authoritative step** — `combat_sim.step_combat_frame` + `combat_engine` are
    meant to converge on “one deterministic tick” the server runs and clients only
    display + send commands. Keeping rules out of pygame input code supports that.

Practical direction
-------------------
  - Prefer **parameters** (hooks, scenario struct, side lists) over new special cases.
  - Move **pure rules** into `combat_engine.py` / `combat_sim.py`; keep **I/O and
    presentation** in scene modules.
  - When adding MP, thread **commands in → state out** rather than duplicating combat
    logic on the client.

Engine architecture (current)
-----------------------------
The monolithic `demo_game.run()` has been replaced by a scene-based engine:

  - `engine.py`         — pygame init, main loop, SceneManager orchestration
  - `draw.py`           — shared rendering primitives (ships, fog, UI chrome)
  - `scenes.py`         — RunContext, Scene protocol, SceneManager
  - `game_state.py`     — GameState dataclass tree (all mutable runtime state)

Scene files (one per game phase):
  - `scene_config.py`      — title / menu
  - `scene_loadouts.py`    — fleet builder before combat
  - `scene_combat.py`      — the core gameplay loop
  - `scene_debrief.py`     — between-round store / upgrades
  - `scene_gameover.py`    — end-of-run overlay
  - `scene_mp_hub.py`      — multiplayer lobby browser
  - `scene_mp_lobby.py`    — in-lobby settings, chat, ready/start
  - `scene_bg_editor.py`   — battlegroup preset editor

Combat simulation (unchanged):
  - `combat_engine.py`     — movement, fog, spawning, engagement
  - `combat_ordnance.py`   — weapons, PD, VFX
  - `combat_sim.py`        — per-frame tick (`step_combat_frame`)
  - `combat_mp.py`         — MP command application
  - `combat_snapshot.py`   — state serialization for net sync
  - `combat_constants.py`  — tuning values
  - `combat_math.py`       — geometry helpers

Entity types and utility functions still live in `demo_game.py` (retained as
the data/entity library; its `run()` function is no longer the active entry
point).

This file is intent only; behavior is defined by code.
